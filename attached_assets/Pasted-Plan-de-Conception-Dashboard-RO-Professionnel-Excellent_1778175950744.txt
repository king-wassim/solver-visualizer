Plan de Conception — Dashboard RO Professionnel
Excellente démarche de commencer par la stratégie avant le code. Voici mon analyse complète en tant que Product Manager / UX Designer, avec un cadrage polyvalent (pas figé sur les 6 exercices actuels) pour que ton dashboard puisse accueillir n'importe quel problème de Recherche Opérationnelle.

Le pivot clé : passer d'une app spécifique aux TPs à une plateforme générique de résolution PL dans laquelle les TPs ne sont que des cas pré-chargés.

3. Charte graphique
Je recommande Minimalisme type Vercel/Linear plutôt que Neobrutalisme. Justification : un dashboard analytique académique doit privilégier la lisibilité des données sur la personnalité visuelle. Le neobrutalisme (bordures épaisses, ombres dures) fatigue l'œil sur des sessions longues et concurrence visuellement les graphiques.
Palette
Mode sombre (par défaut, cohérent avec ton existant)

Fond : #09090B (zinc-950) — plus neutre que ton #0D1117 actuel, moins bleuté
Surface : #18181B (zinc-900)
Surface élevée : #27272A (zinc-800)
Bordure : #27272A / #3F3F46
Texte primaire : #FAFAFA
Texte secondaire : #A1A1AA

Mode clair

Fond : #FFFFFF
Surface : #FAFAFA
Bordure : #E4E4E7
Texte : #09090B / #71717A

Accents sémantiques (identiques dans les deux modes, ajustés en saturation)

Primaire : #3B82F6 (blue-500) — actions, liens
Succès : #10B981 (emerald-500) — Optimal, contraintes OK
Warning : #F59E0B (amber-500) — saturation forte
Erreur : #EF4444 (red-500) — Infaisable
Neutre data : palette catégorielle 8 couleurs (Tailwind)

Style de composants
Système : shadcn/ui + Tailwind

Bordures fines (1px), rayons modérés (6-8px)
Ombres subtiles (shadow-sm)
Typographie : Inter pour l'UI, JetBrains Mono pour code/chiffres (cohérent avec ton choix actuel)
Animations courtes (150-200ms), Framer Motion pour transitions de page

Pourquoi ce choix vs alternatives :

Material UI : trop opinioné, "google-esque", lourd
Chakra/Mantine : bons mais moins flexibles que shadcn (qui copie le code dans ton projet → contrôle total)
Neobrutalisme : tendance, mais inadapté à un dashboard data-dense

Procédure de déploiement (gratuit)
Étape 1 — Repository GitHub
1. Créer repo public "ro-dashboard" sur github.com
2. git init, commit, push
3. Ajouter README, LICENSE (MIT), .gitignore (Node)
Étape 2 — Vercel (recommandé)
1. vercel.com → Sign in with GitHub
2. Import Project → sélectionner ro-dashboard
3. Framework preset: Vite (auto-détecté)
4. Build command: npm run build
5. Output directory: dist
6. Deploy → URL automatique : ro-dashboard.vercel.app
Avantages Vercel vs Netlify pour ce projet :

Preview deployments par PR (chaque branche = URL de test → utile pour montrer au prof avant de merger)
Edge network mondial gratuit
Analytics intégrés (gratuit tier)
Build cache plus rapide

Étape 3 — Domaine custom (optionnel)

Vercel offre *.vercel.app gratuit
Domaine .tn ou .com ~10€/an si tu veux du sérieux

Étape 4 — CI/CD automatique

Chaque git push sur main → redéploiement automatique
Pas de configuration nécessaire (Vercel détecte tout)

Étape 5 — Performance & SEO

Lighthouse score visé : >95 (faisable avec Vite + WASM lazy-loaded)
Meta tags OpenGraph pour partage social
robots.txt + sitemap.xml


5. Phasage du projet (4 étapes)
Phase 1 — Fondations (Semaine 1)
Objectif : un squelette navigable et déployé

Setup Vite + TS + Tailwind + shadcn
React Router + layout principal (sidebar + topbar + content)
Theme provider (dark/light)
Page d'accueil statique avec KPIs mockés
Premier déploiement Vercel
Livrable : URL Vercel avec navigation entre pages vides mais stylées

Phase 2 — Moteur de résolution générique (Semaine 2)
Objectif : résoudre n'importe quel PL côté client

Intégration glpk.js
Définition du schéma JSON d'un modèle LP (Zod schema)
Fonction solve(model) → result typée
Charger les 6 TPs comme fichiers JSON dans /public/models/
Page "Bibliothèque" qui liste et charge ces modèles
Livrable : on peut résoudre les 6 TPs existants sans aucun code spécifique

Phase 3 — UI riche (Semaine 3)
Objectif : visualisations et éditeur

Éditeur de modèle (mode formulaire + JSON)
Page Résultats avec onglets (Solution / Contraintes / Visu)
Composants de visualisation génériques :

Région réalisable (auto-détectée si 2 variables)
Bar chart variables (n'importe quel nb de vars)
Heatmap + Sankey (auto-détectés si problème de transport)
Radar utilisation contraintes


Export CSV/JSON
Livrable : parité fonctionnelle complète avec ton Streamlit + meilleure UX

Phase 4 — Différenciateurs & polish (Semaine 4)
Objectif : dépasser le prototype

Comparateur de scénarios (what-if analysis)
Analyse de sensibilité (coûts réduits, prix duaux)
Mode présentation (fullscreen pour soutenance)
Animations Framer Motion
Tests unitaires sur le solveur
Documentation utilisateur (page /docs)
Optimisations bundle (code splitting, lazy load WASM)
Livrable : produit de niveau portfolio professionnel


Ce qui rend ce plan "polyvalent et passe-partout"

Schéma JSON unifié pour tout PL → ajouter un TP = ajouter un fichier JSON, jamais du code
Visualisations auto-détectées selon la structure du modèle (2 vars → région, transport → Sankey)
Solveur WebAssembly → fonctionne pour PL continu, entier, mixte sans changer d'archi
Pas de backend → coût zéro, déploiement instantané
Design system (shadcn) → réutilisable pour tes futurs projets
